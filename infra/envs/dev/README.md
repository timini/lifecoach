# infra/envs/dev

Terraform composition for the dev environment.

## First-time setup

```bash
# From repo root, one time:
just bootstrap env=dev
```

This creates the project, state bucket, and writes `backend.hcl` + `terraform.tfvars` here (both gitignored).

## Day-to-day

```bash
just tf-init dev   # once, or after backend changes
just tf-plan dev
just tf-apply dev
```

## Files

| File | Committed? | Purpose |
|---|---|---|
| `backend.tf` | yes | Declares GCS backend (bucket wired via `-backend-config`) |
| `providers.tf` | yes | google / google-beta providers |
| `variables.tf` | yes | Variable declarations |
| `main.tf` | yes | Module composition for this env |
| `terraform.tfvars.example` | yes | Reference for what tfvars should contain |
| `backend.hcl.example` | yes | Reference for what backend.hcl should contain |
| `terraform.tfvars` | **no** | Generated by bootstrap.sh |
| `backend.hcl` | **no** | Generated by bootstrap.sh |
| `.terraform/` | no | Terraform internals |
| `.terraform.lock.hcl` | yes (when added by init) | Provider version lock |

## E2E test user

The Playwright chat-persistence spec (`apps/web/e2e/chat-persistence.spec.ts`) needs a stable Email/Password user in Firebase Auth. Provision it once per environment:

```bash
just provision-e2e-user dev
```

That creates `e2e-test@lifecoach.invalid` (or rotates its password if it already exists), tags the user with `{ e2e: true }`, and stores the password in Secret Manager as `E2E_TEST_PASSWORD`.

To run the spec against the deployed environment:

```bash
just e2e-deployed dev
```

`infra/modules/firebase-auth` already enables the email provider with `password_required = false`, which permits **both** password sign-in (used by the e2e harness) and email-link sign-in (used by the regular UI). No Terraform change is needed for this.
